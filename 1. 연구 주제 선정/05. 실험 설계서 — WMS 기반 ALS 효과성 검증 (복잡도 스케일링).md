# 실험 설계서 v3: WMS 기반 ALS 효과성 검증 — 복잡도 스케일링

> **목적**: 비즈니스 규칙의 복잡도가 증가할 때, ALS 구조화 형식의 효과가 어떻게 달라지는지를 실증한다.
> **핵심 가설**: 규칙이 단순할 때는 자연어 설계문서와 ALS의 차이가 미미하지만, 복잡도가 높아지면 ALS의 구조적 이점이 유의미하게 발현된다.
> **실험 도구**: Claude Code CLI (에이전트 모드)
> **실험 도메인**: WMS (창고관리시스템)
> **LLM**: Claude Sonnet 4.5 (Anthropic, `claude-sonnet-4-5-20250929`)

---

## 1. 연구 배경 — ver2 실험에서의 발견

### 1.1 ver2 실험 결과 요약

ver2에서는 단일 턴 API(실험 1)와 AI 코딩 에이전트(실험 2)로 A(바이브코딩)/B(자연어 설계)/C(ALS) 3그룹을 비교했다.

| 메트릭 | Exp1: A | Exp1: B | Exp1: C | Exp2: A | Exp2: B | Exp2: C |
|--------|---------|---------|---------|---------|---------|---------|
| LCR    | 90.3%   | 100%    | 100%    | 90.4%   | 98.8%   | 99.6%   |
| CDR    | 100%    | 100%    | 100%    | 45.0%   | 90.0%   | 100%    |
| SDC    | 16.8    | 5.4     | 5.2     | 30.6    | 0.2     | 2.3     |

### 1.2 핵심 발견과 한계

1. **설계 문서의 유무(A vs B/C)가 압도적 변수**: 모든 메트릭에서 A ≪ B ≈ C
2. **ALS vs 자연어 설계의 차이는 CDR에서만 유의미**: C(100%) > B(90.0%) > A(45.0%)
3. **LCR/SDC에서 B ≈ C, 심지어 SDC에서 B < C**: ALS가 자연어 설계보다 낫다는 증거 불충분

### 1.3 ver3의 가설

ver2에서 B ≈ C인 이유는 **규칙의 복잡도가 현재 LLM의 처리 능력 대비 낮았기 때문**이라는 가설을 세운다.

- ver2의 규칙: 48항목, 4개 태스크, 규칙 간 독립적, 단일 조건 분기 위주
- 실무 WMS: 수백 개 규칙, 품목별/계절별/거래처별 조건 분기, 규칙 간 상호의존, 교차 도메인 연쇄

ver2의 규칙은 Claude Sonnet 4.5에게 **"너무 쉬운 시험"**이었을 가능성이 높다. 복잡도를 실무 수준에 가깝게 올리면 자연어 설계문서의 한계가 드러나고, ALS 구조화의 효과가 발현될 것으로 예상한다.

## 3. 실험 설계 개요

### 3.1 실험 모드: 에이전트 전용

ver2에서 단일 턴 API 실험은 다음과 같은 한계가 확인되어 **ver3에서는 에이전트 모드만 수행한다**.

| 단일 턴의 한계 | 설명 |
|--------------|------|
| 비현실적 시나리오 | 실무에서 단일 턴으로 바이브코딩하는 개발자는 없음 |
| CDR 천장효과 | 누적 컨텍스트가 프롬프트에 포함되어 세 그룹 모두 100% 달성 |
| 복잡도 스케일링 불가 | 프롬프트 길이 제한으로 100+ 규칙을 한 번에 전달하기 어려움 |

에이전트 모드는 파일 시스템에서 문서를 **능동적으로 검색**해야 하므로, 문서의 구조화 수준이 검색 효율과 규칙 참조 정확도에 직접 영향을 미친다. 이것이 ALS 구조화 효과가 발현되는 핵심 메커니즘이다.

### 3.2 비교군 구성 (3그룹, ver2와 동일)

```
[공통] WMS 요구사항 정의서 (동일)
  │
  ├── Group A (바이브코딩 대조군):
  │     요구사항 정의서 + 태스크 지시
  │     (설계 문서 없음 — 일반적인 바이브코딩 시나리오)
  │
  ├── Group B (자연어 설계문서):
  │     요구사항 정의서 + 자연어 설계 문서 + 태스크 지시
  │     (DDL, API 스펙, 비즈니스 규칙을 산문 형태로 제공)
  │
  └── Group C (ALS):
        요구사항 정의서 + ALS 문서 + 태스크 지시
        (동일 정보를 ALS 형식으로 구조화하여 제공)
```

### 3.3 복잡도 스케일링 — 2단계

| 항목 | 기본 복잡도 | 확장 복잡도 |
|------|---------------|---------------|
| **규칙 수** | 48항목 | ~100항목 (2배+) |
| **조건 분기** | 단일 if | 3~5중 중첩 (품목별 × 타입별 × 계절별) |
| **규칙 간 의존성** | 없음 (태스크 독립) | 동일 도메인 내 + 교차 도메인 |
| **예외 처리** | 없음 또는 1단계 | 예외의 예외 (2~3단계 중첩) |
| **상태 전이** | 단순 (2~3상태) | 조건부 상태 전이 (5~7상태) |
| **시간 의존성** | 없음 | 계절별, 유통기한 잔여율 기반 |
| **연쇄 트리거** | 없음 | 다단계 (출고→안전재고→자동발주) |
| **충돌 시나리오** | 4개 (명시적) | 8개 (명시적 + 묵시적 + 교차도메인) |
| **체감 복잡도** | 1x (기준) | **~3x** |
| **데이터** | ver2 재활용 | **신규 실험** |

## 4. 확장 복잡도 — 규칙 설계

### 4.1 설계 원칙

확장 규칙은 실무 WMS에서 흔히 존재하는 복잡성을 추가한다. 추가되는 복잡성의 유형은 다음 세 가지이다:

1. **조건 분기 중첩**: 하나의 규칙이 품목 카테고리, 거래 유형, 계절, 수입 여부 등 여러 조건에 따라 다르게 적용됨
2. **규칙 간 의존성**: 한 태스크의 결과가 다른 태스크의 조건에 영향을 미침 (예: 입고 결과 → 안전재고 재계산 → 자동 발주 트리거)
3. **묵시적 제약**: 명시적으로 기술되지 않았지만 도메인 상식에서 유도되는 불변 조건 (예: 위험물 혼적 금지)

### 4.2 추가 엔티티 및 스키마 확장

확장 복잡도에서는 19개 테이블로 구성하며, 다음 엔티티/컬럼을 포함한다:

| 엔티티/컬럼 | 설명 | 관련 규칙 |
|------------|------|----------|
| `products.category` | 상품 카테고리 (GENERAL, FRESH, HAZMAT, HIGH_VALUE) | 카테고리별 규칙 분기 |
| `products.storage_type` | 보관 유형 (AMBIENT, COLD, FROZEN) | 로케이션 호환성 체크 |
| `products.min_remaining_shelf_life_pct` | 최소 잔여 유통기한 비율 (입고 거부 기준) | 입고 시 유통기한 검증 강화 |
| `locations.storage_type` | 로케이션 보관 유형 | 상품-로케이션 호환성 |
| `locations.zone` | 구역 (RECEIVING, STORAGE, SHIPPING, HAZMAT) | 위험물 전용 구역 |
| `purchase_orders.po_type` | 발주 유형 (NORMAL, URGENT, IMPORT) | 발주 유형별 규칙 분기 |
| `purchase_orders.origin_country` | 원산지 (수입품 판별) | 수입품 관련 규칙 |
| `suppliers` | 공급업체 마스터 | 공급업체 실적/페널티 |
| `supplier_penalties` | 공급업체 페널티 이력 | 입고 불량 시 페널티 부과 |
| `safety_stock_rules` | 안전재고 기준 | 자동 발주 트리거 |
| `auto_reorder_logs` | 자동 재발주 이력 | 교차 도메인 연쇄 |
| `seasonal_config` | 계절 설정 (성수기/비수기) | 시간 의존적 규칙 |

### 4.3 Task 1 확장: 입고 처리 (Level 2)

**ver2 규칙 (유지):**
- 기본 입고 흐름 (PO 대조, 2단계 프로세스, 재고 반영 시점)
- 초과입고 10% 제한, 유통기한 필수 체크, received_qty 누적 등

**Level 2 추가 규칙:**

**(a) 품목 카테고리별 초과입고 허용률 차등**
```
카테고리별 초과 허용률:
├── GENERAL (일반):     10%
├── FRESH (신선식품):    5%  (폐기 리스크 때문)
├── HAZMAT (위험물):     0%  (안전 규정상 초과 불가)
└── HIGH_VALUE (고가품): 3%  (재고 비용 관리)
```

**(b) 발주 유형별 오버라이드**
```
PO 유형별 초과 허용률 가중치:
├── NORMAL:  카테고리 기준 그대로
├── URGENT:  카테고리 기준 × 2 (긴급이므로 유연하게)
└── IMPORT:  카테고리 기준 × 1.5 (통관 지연 감안)
    └── 단, HAZMAT은 IMPORT라도 0% 유지 (안전 규정 예외 없음)
```

**(c) 계절별 가중치**
```
성수기 판정 (seasonal_config 테이블 참조):
├── 성수기: 초과 허용률 × 1.5 (물량 증가 대비)
└── 비수기: 초과 허용률 그대로
    └── 단, HAZMAT은 성수기라도 0% 유지
```

**(d) 유통기한 잔여율 기반 입고 거부**
```
유통기한 관리 상품 입고 시:
├── 잔여 유통기한 < 30%: 입고 거부 (사유: 유통기한 부족)
├── 잔여 유통기한 30~50%: 경고 발생 + 관리자 승인 필요
└── 잔여 유통기한 > 50%: 정상 입고
    잔여율 계산: (expiry_date - today) / (expiry_date - manufacture_date) × 100
```

**(e) 보관 유형 호환성 체크**
```
입고 시 로케이션 배정:
├── products.storage_type과 locations.storage_type이 일치해야 함
├── FROZEN 상품 → AMBIENT 로케이션 배정 시 거부
├── HAZMAT 상품 → HAZMAT zone 로케이션만 허용
└── COLD 상품 → COLD 또는 FROZEN 로케이션 허용 (상위 호환)
```

**(f) 입고 거부 시 공급업체 페널티**
```
초과입고 거부 또는 유통기한 부족 거부 발생 시:
├── supplier_penalties 테이블에 페널티 이력 기록
├── 페널티 유형: OVER_DELIVERY, SHORT_SHELF_LIFE
└── 동일 공급업체 최근 30일 페널티 3회 이상 → 해당 공급업체 PO 자동 보류(hold)
```

**확장 Task 1 예상 채점 항목: ~25항목**

### 4.4 Task 2 확장: 출고 처리 (Level 2)

**ver2 규칙 (유지):**
- FIFO/FEFO, 부분출고, 백오더 등 기본 출고 규칙

**Level 2 추가 규칙:**

**(a) 위험물 출고 제약**
```
HAZMAT 상품 출고 시:
├── HAZMAT zone 로케이션에서만 피킹
├── 1회 출고당 최대 수량 제한 (products.max_pick_qty 참조)
├── 동일 출고 지시서에 HAZMAT + FRESH 상품이 공존하면 분리 출고 필수
└── 분리 출고: 별도의 shipment_order로 분할 생성
```

**(b) 유통기한 만료 임박 상품 우선 출고**
```
FEFO 확장:
├── 잔여 유통기한 < 30%: 최우선 출고 (FEFO 내에서 추가 우선순위)
├── 잔여 유통기한 < 10%: 출고 불가 (폐기 대상으로 자동 전환)
│   └── 해당 재고 상태를 EXPIRED로 변경, 출고 대상에서 영구 제외
└── 잔여 유통기한 만료 (expiry_date < today): ver2와 동일하게 제외
```

**(c) 부분출고 의사결정 트리**
```
재고 부족 시 처리 (기존: 무조건 부분출고+백오더):
├── 가용 재고 ≥ 요청의 70%: 부분출고 + 백오더
├── 가용 재고 < 요청의 70% 그리고 ≥ 30%: 부분출고 + 백오더 + 긴급발주 트리거
├── 가용 재고 < 요청의 30%: 전량 백오더 (부분출고 안 함)
└── 가용 재고 = 0: 전량 백오더
    긴급발주 트리거 시 auto_reorder_logs에 기록
```

**(d) 안전재고 연계 (교차 도메인)**
```
출고 완료 후 연쇄 체크:
├── 해당 상품의 전체 가용 재고 합산
├── safety_stock_rules.min_qty 이하이면:
│   ├── auto_reorder_logs에 자동 발주 요청 기록
│   └── 발주 수량 = safety_stock_rules.reorder_qty
└── 안전재고 이상이면 아무 것도 하지 않음
```

**(e) 보관 유형 간 출고 시 경고**
```
피킹한 로케이션의 storage_type과 출고 대상 상품의 storage_type이 다른 경우:
└── 이는 재고 이동 오류의 후유증일 수 있으므로 경고 로그를 기록
    (출고 자체는 차단하지 않으나, 이상 로그를 남김)
```

**확장 Task 2 예상 채점 항목: ~28항목**

### 4.5 Task 3 확장: 재고 이동 (Level 2)

**ver2 규칙 (유지):**
- 단일 트랜잭션, 용량 체크, 동일 로케이션 거부, 이력 기록 등

**Level 2 추가 규칙:**

**(a) 보관 유형 호환성 검증**
```
이동 시 도착지 로케이션 호환성:
├── products.storage_type과 도착지 locations.storage_type 일치 필수
├── FROZEN → AMBIENT: 거부 (품질 위험)
├── COLD → AMBIENT: 거부
├── HAZMAT → 비-HAZMAT zone: 거부 (안전 규정)
└── AMBIENT → COLD/FROZEN: 허용 (상위 호환)
```

**(b) 위험물 혼적 금지**
```
도착지 로케이션에 이미 적재된 상품과의 호환성:
├── HAZMAT 상품 이동 시, 도착지에 비-HAZMAT 상품이 있으면 거부
├── 비-HAZMAT 상품 이동 시, 도착지에 HAZMAT 상품이 있으면 거부
└── 즉, HAZMAT과 비-HAZMAT의 동일 로케이션 혼적 전면 금지
```

**(c) 유통기한 임박 상품 이동 제한**
```
유통기한 관리 상품 이동 시:
├── 잔여 유통기한 < 10%: SHIPPING zone으로만 이동 허용 (즉시 출고 목적)
├── 잔여 유통기한 만료: 이동 불가 (폐기 절차로 전환 필요)
└── 그 외: 정상 이동
```

**(d) 대량 이동 승인**
```
이동 수량이 출발지 재고의 80% 이상인 경우:
├── 관리자 승인 필요 (approval_status = pending)
├── 승인 없이 즉시 이동 불가
└── 이유: 대량 이동은 물류 운영에 큰 영향을 미치므로 검증 필요
```

**(e) 이동 후 안전재고 체크 (교차 도메인)**
```
이동 완료 후:
├── 출발지 로케이션이 속한 구역(STORAGE)의 해당 상품 총 재고 확인
├── safety_stock_rules.min_qty 이하이면 경고 로그 기록
│   └── auto_reorder_logs에 기록 (출고 연쇄와 동일 메커니즘)
└── 안전재고 이상이면 아무 것도 하지 않음
```

**확장 Task 3 예상 채점 항목: ~20항목**

### 4.6 Task 4 확장: 재고 실사 및 조정 (Level 2)

**ver2 규칙 (유지):**
- 사유 필수, 5% 자동/수동 승인, 삭제 불가, 수정 불가 등

**Level 2 추가 규칙:**

**(a) 카테고리별 차등 자동승인 임계치**
```
자동 승인 기준 (기존: 일률 ±5%):
├── GENERAL: ±5%
├── FRESH: ±3%    (신선식품은 더 엄격한 관리)
├── HAZMAT: ±1%   (위험물은 미세 차이도 추적)
└── HIGH_VALUE: ±2% (고가품은 비용 영향이 큼)
```

**(b) 연속 조정 감시**
```
동일 로케이션 + 동일 상품에 대해 최근 7일 내 조정이 2회 이상이면:
├── 자동 승인 대상이라도 무조건 관리자 승인 필요
├── 사유에 "연속 조정 감시 대상" 자동 태그 추가
└── 이유: 반복적 차이는 구조적 문제(도난, 시스템 오류 등)를 의심해야 함
```

**(c) 조정 후 안전재고 연계 (교차 도메인)**
```
조정이 반영된 후 (approved 또는 auto_approved):
├── 해당 상품의 전체 재고 합산
├── safety_stock_rules.min_qty 이하이면 자동 발주 트리거
│   └── auto_reorder_logs에 기록
└── 안전재고 이상이면 아무 것도 하지 않음
```

**(d) 고가품 전수 검증**
```
HIGH_VALUE 카테고리 상품의 조정:
├── 차이가 0이 아닌 모든 경우 관리자 승인 필요 (자동 승인 없음)
├── 조정 반영 시 별도 감사 로그(audit_log) 추가 기록
│   └── 기록 내용: 조정 시각, 승인자, 조정 전후 수량, 금액 환산
└── 1건이라도 차이가 있으면 해당 로케이션 전체 재실사 권고 알림
```

**(e) 실사 중 입출고 동결**
```
실사가 진행 중(in_progress)인 로케이션:
├── 해당 로케이션에 대한 입고 배정 불가
├── 해당 로케이션에서의 출고 피킹 불가
├── 해당 로케이션으로/에서의 재고 이동 불가
└── 실사 완료(조정 반영 또는 일치 확인) 시 동결 해제
```

**확장 Task 4 예상 채점 항목: ~25항목**

## 5. 측정 메트릭

### 5.1 기존 메트릭과의 관계

LLM 코드 생성 평가에서 널리 사용되는 메트릭(Pass@k, CodeBLEU, SWE-bench 등)은 알고리즘 문제나 함수 단위 태스크를 대상으로 한다. **도메인 고유의 비즈니스 규칙 준수, 설계 충돌 탐지, 스키마 할루시네이션 억제**를 측정하는 표준 메트릭은 존재하지 않는다. 이에 본 연구는 다음 세 가지 메트릭을 자체 정의하여 사용한다.

### 5.2 메트릭 정의

**RQ1, RQ4 → Logic Compliance Rate (LCR)**
```
LCR = (준수한 규칙 수 / 전체 규칙 수) × 100%
```
- 기본: 48항목 체크리스트 기준
- 확장: ~98항목 체크리스트 기준 (Task 5 제외)
- 태스크별로 사전 정의된 체크리스트 기반, Yes/No 판정

**RQ2, RQ4 → Conflict Detection Rate (CDR)**
```
CDR = (탐지 성공 수 / 전체 충돌 시나리오 수) × 100%
```
- 기본: 4개 시나리오
- 확장: 8개 시나리오
- 탐지 성공 기준: AI가 충돌을 명시적으로 언급하고 거부/경고/대안 제시

**RQ3, RQ4 → Schema Deviation Count (SDC)**
```
SDC = 태스크당 스키마 일탈 발생 건수 (낮을수록 좋음)
```
- 7가지 유형: D1(필드명 변경), D2(타입 변경), D3(필드 누락), D4(임의 추가), D5(상태값 변경), D6(테이블명 변경), D7(제약조건 누락)
- 확장 복잡도에서는 DDL이 확장되므로 일탈 가능성이 더 높아짐

### 5.3 RQ4 분석: 복잡도 × 그룹 상호작용

RQ4는 기본 복잡도와 확장 복잡도에서 **구조화 효과(C - B)의 크기**를 비교한다.

```
구조화 효과(기본) = C_basic - B_basic
구조화 효과(확장) = C_ext - B_ext

RQ4 검증: 구조화 효과(확장) > 구조화 효과(기본) 인가?
```

만약 확장 복잡도에서 구조화 효과가 유의미하게 커진다면, "복잡도가 높을 때 구조화가 필수적"이라는 가설이 지지된다.

## 8. 예상 결과 및 시나리오 분석

### 8.1 가설 지지 시나리오 (이상적)

| 메트릭 | 기본 (C-B) | 확장 (C-B) | 해석 |
|--------|--------------|---------------|------|
| LCR    | +0.8%p       | +5~15%p       | 복잡도 증가 시 ALS 구조화가 규칙 누락 방지에 유의미 |
| CDR    | +10.0%p      | +20~40%p      | 복잡한 충돌 시나리오에서 ALS의 앵커링 효과 증대 |
| SDC    | -2.1건       | -5~10건       | 스키마가 복잡해지면 ALS의 명시적 제약이 일탈 억제 |

### 8.2 부분 지지 시나리오

- CDR에서만 확장 복잡도 효과 발현: 구조화는 충돌 탐지에 특화된 효과
- LCR에서만 효과 발현: 구조화는 규칙 누락 방지에 효과적이지만 충돌 탐지에는 추가 효과 없음

### 8.3 가설 기각 시나리오

- 확장 복잡도에서도 B ≈ C: **"현재 LLM은 자연어 설계문서만으로 충분하며, 구조화의 추가 효과는 없다"**
- 이 경우에도 **"복잡도를 2배/3배로 올려도 구조화 효과가 없다"는 negative result 자체가 유의미한 학술적 기여**가 된다.

## 9. Threats to Validity

| 위협 유형 | 내용 | 완화 방안 |
|-----------|------|----------|
| **Internal: 복잡도 조작** | Level 2 규칙이 ALS에 유리하게 설계되었을 가능성 | 규칙은 실무 WMS 도메인에서 유래하며, B와 C에 동일 정보 제공 |
| **Internal: 복잡도 간 비교** | 기본과 확장의 규칙 수/유형이 달라 직접 비교의 한계 | 구조화 효과(C-B)를 비교하여 복잡도 간 상대적 변화 측정 |
| **Internal: 단일 평가자** | 주관적 판단 개입 가능 | 체크리스트 기반 기계적 채점 + SDC는 DDL 대조 자동화 가능 |
| **Internal: 문서 분량 차이** | B(자연어)와 C(ALS)의 분량 차이가 확장 복잡도에서 더 커질 수 있음 | 정보량 동일성 검증 + ablation study 가능성 언급 |
| **External: 단일 도메인** | WMS만으로 일반화 한계 | Threats에 명시 + Future Work에 다중 도메인 제안 |
| **External: 단일 LLM** | Claude Sonnet 4.5만 사용 | 모델 간 차이는 향후 연구 과제 |
| **Construct: 메트릭 자체 정의** | LCR/CDR/SDC가 표준 메트릭이 아님 | 기존 메트릭과의 관계 명시 + 자체 정의의 정당성 논증 |
| **Reliability: 표본 크기** | $n=5$로 통계적 유의성 한계 | 효과 크기(effect size) 보고 + 반복 결과의 일관성 분석 |
