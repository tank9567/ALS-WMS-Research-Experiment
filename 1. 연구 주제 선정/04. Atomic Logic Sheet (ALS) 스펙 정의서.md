# Atomic Logic Sheet (ALS) 스펙 정의서 v0.1

> 본 문서는 H-BLI & Anchoring 프레임워크의 핵심 산출물인 Atomic Logic Sheet의 형식 스펙을 정의한다.
> 논문 Section 3 "Proposed Framework"의 기초 자료로 사용한다.

---

## 1. ALS란 무엇인가

**Atomic Logic Sheet(ALS)**는 소프트웨어 프로젝트의 비즈니스 로직을 LLM이 참조하기 최적의 형태로 구조화한 마크다운 기반 문서이다.

### 1.1 설계 원칙

| 원칙 | 설명 |
|------|------|
| **Atomicity (원자성)** | 하나의 ALS는 하나의 독립된 비즈니스 규칙 또는 제약조건만 다룬다 |
| **Self-containedness (자기완결성)** | 외부 문서를 참조하지 않아도 해당 규칙을 완전히 이해할 수 있어야 한다 |
| **Machine-readability (기계 가독성)** | LLM의 컨텍스트 윈도우에 효율적으로 주입 가능한 구조여야 한다 |
| **Human-readability (인간 가독성)** | 개발자가 리뷰하고 승인할 수 있을 만큼 직관적이어야 한다 |
| **Immutability by default (기본 불변성)** | 명시적인 변경 승인 없이는 수정되지 않는다 (Logic Anchor 역할) |

### 1.2 기존 접근법과의 차이

| 비교 대상 | 한계 | ALS의 개선점 |
|-----------|------|-------------|
| 자연어 설계 문서 | 모호성, 컨텍스트 윈도우 낭비 | 구조화된 필드로 모호성 제거 |
| 코드 주석 | 코드에 종속, 전체 그림 부재 | 코드 독립적, 계층 구조 제공 |
| Cursor Rules / .cursorrules | 프로젝트 전체 규칙에 한정, 개별 로직 단위 미지원 | 개별 비즈니스 규칙 단위로 원자화 |
| System Prompt 직접 작성 | 비표준, 재사용 불가, 충돌 검증 불가 | 표준 스키마로 검증 가능 |

## 2. ALS 스키마 정의

### 2.1 계층 구조 (H-BLI Hierarchy)

ALS는 3계층으로 구성된다:

```
Level 0: Project Context (프로젝트 컨텍스트)
  └─ Level 1: Domain Rule (도메인 규칙)
       └─ Level 2: Implementation Constraint (구현 제약조건)
```

| 계층 | 역할 | 예시 |
|------|------|------|
| **L0 - Project Context** | 프로젝트 전체에 적용되는 기술적 맥락 (DB 스키마, 기술 스택, 공통 규칙) | DDL 정의, API 공통 응답 포맷 |
| **L1 - Domain Rule** | 비즈니스 도메인에서 유래하는 업무 규칙 | "환자의 약물 알레르기가 있으면 해당 약물 처방 차단" |
| **L2 - Implementation Constraint** | L1을 코드로 구현할 때의 구체적 제약 | "알레르기 체크는 처방 API의 validation 단계에서 수행, HTTP 409 반환" |

### 2.2 단일 ALS 문서 스키마

```markdown
---
als_id: [고유 식별자, 예: ALS-RX-001]
als_version: [버전, 예: 1.0]
level: [0 | 1 | 2]
domain: [도메인 분류, 예: prescription]
status: [draft | active | deprecated]
depends_on: [의존하는 다른 ALS ID 목록]
created: [작성일]
last_reviewed: [마지막 리뷰일]
owner: [작성자/책임자]
---

# [ALS 제목]

## Rule (규칙)
> 이 규칙이 무엇인지 한 문장으로 명확히 기술한다.
> 형식: "WHEN [조건] THEN [행동] BECAUSE [근거]"

## Context (맥락)
- 이 규칙이 왜 존재하는지 (비즈니스 배경)
- 관련된 이해관계자 또는 법적 근거

## Constraints (제약조건)
- [ ] 제약조건 1
- [ ] 제약조건 2

## Data Schema (관련 데이터 구조)
관련 테이블/필드의 DDL 또는 스키마 발췌 (L0에서 가져오거나 직접 명시)

## Valid Examples (정상 케이스)
입력 → 기대 출력 형태로 구체적 예시 제공

## Invalid Examples (위반 케이스)
규칙을 위반하는 입력 → 기대되는 에러/차단 행동

## Anti-patterns (금지 패턴)
이 규칙을 구현할 때 절대 하면 안 되는 것들
```

### 2.3 필드별 상세 설명

| 필드 | 필수 여부 | 설명 |
|------|-----------|------|
| `als_id` | **필수** | 프로젝트 내 고유 식별자. `ALS-{DOMAIN}-{SEQ}` 형식 권장 |
| `als_version` | **필수** | Semantic versioning. Logic Anchor 변경 추적에 사용 |
| `level` | **필수** | H-BLI 계층 (0, 1, 2) |
| `domain` | **필수** | 비즈니스 도메인 분류 태그 |
| `status` | **필수** | `active`인 ALS만 Logic Anchor로 기능함 |
| `depends_on` | 선택 | 이 규칙이 전제하는 다른 ALS. 충돌 탐지 시 참조 |
| `Rule` | **필수** | WHEN-THEN-BECAUSE 형식의 핵심 규칙 |
| `Context` | 권장 | 규칙의 존재 이유 (없으면 LLM이 임의 해석할 위험) |
| `Constraints` | **필수** | 체크리스트 형태의 구현 제약조건 |
| `Data Schema` | L0, L2에서 필수 | 관련 DDL 또는 데이터 구조 |
| `Valid/Invalid Examples` | **필수** | LLM이 규칙을 정확히 이해했는지 검증하는 기준 |
| `Anti-patterns` | 권장 | 할루시네이션 방지를 위한 명시적 금지 사항 |

## 3. ALS 적용 예시: 가상 헬스케어 프로젝트 "MediGuard"

> MediGuard: 환자의 처방전을 관리하고, 약물 상호작용 및 알레르기를 자동 검증하는 시스템

### 예시 1: L0 — 프로젝트 컨텍스트

```markdown
---
als_id: ALS-CORE-001
als_version: 1.0
level: 0
domain: core
status: active
depends_on: []
created: 2026-02-16
last_reviewed: 2026-02-16
owner: 차차
---

# MediGuard 데이터베이스 스키마 — 환자 및 처방 테이블

## Rule
> WHEN MediGuard의 어떤 모듈이든 환자/처방 데이터에 접근할 때
> THEN 반드시 아래 정의된 스키마를 기준으로 구현한다
> BECAUSE 스키마와 다른 필드명이나 타입을 사용하면 런타임 에러 및 데이터 정합성 오류가 발생한다

## Data Schema

CREATE TABLE patients (
    patient_id    UUID PRIMARY KEY,
    name          VARCHAR(100) NOT NULL,
    date_of_birth DATE NOT NULL,
    allergies     TEXT[],           -- 알레르기 약물 성분명 배열
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE prescriptions (
    rx_id         UUID PRIMARY KEY,
    patient_id    UUID REFERENCES patients(patient_id),
    drug_code     VARCHAR(20) NOT NULL,   -- KD코드
    drug_name     VARCHAR(200) NOT NULL,
    dosage        VARCHAR(100) NOT NULL,
    ingredients   TEXT[],                  -- 약물 성분명 배열
    prescribed_at TIMESTAMPTZ DEFAULT NOW(),
    status        VARCHAR(20) DEFAULT 'pending'
                  CHECK (status IN ('pending','approved','rejected','dispensed'))
);

## Anti-patterns
- ❌ patients 테이블에 없는 필드(예: `phone`, `address`)를 임의로 추가하지 말 것
- ❌ allergies를 VARCHAR 단일 문자열로 처리하지 말 것 (반드시 TEXT[] 배열)
- ❌ drug_code를 INTEGER로 바꾸지 말 것 (KD코드는 문자 포함)
```

### 예시 2: L1 — 도메인 규칙

```markdown
---
als_id: ALS-RX-001
als_version: 1.0
level: 1
domain: prescription
status: active
depends_on: [ALS-CORE-001]
created: 2026-02-16
last_reviewed: 2026-02-16
owner: 차차
---

# 약물 알레르기 교차검증 규칙

## Rule
> WHEN 새로운 처방(prescription)이 생성될 때
> THEN 처방 약물의 성분(ingredients)과 환자의 알레르기 목록(allergies)을 교차 비교하여,
>      하나라도 일치하면 처방을 차단(reject)한다
> BECAUSE 알레르기 약물 투여는 아나필락시스 등 생명 위협적 부작용을 유발할 수 있다

## Context
- 의료법 시행규칙 제12조: 의약품 처방 시 환자의 알레르기 이력 확인 의무
- 이 규칙은 어떤 상황에서도 예외 없이 적용된다 (의사 override 불가)

## Constraints
- [ ] 비교는 대소문자 무시(case-insensitive)로 수행
- [ ] 부분 일치(partial match)도 차단 대상 (예: "penicillin" ⊂ "amoxicillin/penicillin")
- [ ] 차단 시 어떤 성분이 어떤 알레르기와 충돌했는지 명시해야 함

## Valid Examples

입력: patient.allergies = ["penicillin", "sulfa"]
      drug.ingredients = ["amoxicillin", "clavulanate"]
기대: ✅ 처방 승인 (겹치는 성분 없음)

입력: patient.allergies = ["penicillin", "sulfa"]
      drug.ingredients = ["ampicillin/penicillin", "probenecid"]
기대: ❌ 처방 차단
      사유: "penicillin in drug ingredient 'ampicillin/penicillin'
            matches patient allergy 'penicillin'"

## Invalid Examples (이렇게 구현하면 안 됨)

# ❌ 잘못된 구현: 완전 일치만 체크
if drug.ingredient == patient.allergy:  # "penicillin" == "ampicillin/penicillin" → False → 통과시킴!

# ❌ 잘못된 구현: 차단만 하고 사유 미반환
return {"status": "rejected"}  # 어떤 성분이 문제인지 알 수 없음

## Anti-patterns
- ❌ 의사 권한으로 알레르기 경고를 무시(override)하는 기능을 만들지 말 것
- ❌ 알레르기 체크를 프론트엔드에서만 수행하지 말 것 (반드시 백엔드 API에서 검증)
- ❌ 빈 알레르기 목록(allergies = [])을 "알레르기 없음"이 아닌 "미확인"으로 처리하지 말 것
```

### 예시 3: L2 — 구현 제약조건

```markdown
---
als_id: ALS-RX-001-IMPL
als_version: 1.0
level: 2
domain: prescription
status: active
depends_on: [ALS-RX-001, ALS-CORE-001]
created: 2026-02-16
last_reviewed: 2026-02-16
owner: 차차
---

# 약물 알레르기 교차검증 — API 구현 제약조건

## Rule
> WHEN ALS-RX-001 규칙을 REST API로 구현할 때
> THEN 아래 명시된 엔드포인트, 응답 형식, 에러 코드를 준수한다
> BECAUSE API 계약(contract)이 일관되어야 프론트엔드 및 외부 시스템 연동이 안정적이다

## Constraints
- [ ] 엔드포인트: `POST /api/v1/prescriptions`
- [ ] 알레르기 충돌 시 HTTP 409 Conflict 반환
- [ ] 응답 body에 `conflicts` 배열 포함

## Valid Examples

# 정상 처방 → 201 Created
POST /api/v1/prescriptions
Request Body:
{
  "patient_id": "uuid-123",
  "drug_code": "KD-04521",
  "drug_name": "아목시실린",
  "dosage": "500mg 1일 3회",
  "ingredients": ["amoxicillin", "clavulanate"]
}

Response: 201 Created
{
  "rx_id": "uuid-456",
  "status": "approved"
}

# 알레르기 충돌 → 409 Conflict
POST /api/v1/prescriptions
Request Body:
{
  "patient_id": "uuid-789",       // allergies: ["penicillin"]
  "drug_code": "KD-08832",
  "drug_name": "암피실린",
  "dosage": "250mg 1일 4회",
  "ingredients": ["ampicillin/penicillin"]
}

Response: 409 Conflict
{
  "status": "rejected",
  "reason": "ALLERGY_CONFLICT",
  "conflicts": [
    {
      "ingredient": "ampicillin/penicillin",
      "matched_allergy": "penicillin",
      "match_type": "partial"
    }
  ]
}

## Anti-patterns
- ❌ 알레르기 충돌을 200 OK로 반환하면서 body에만 에러 표시하지 말 것
- ❌ 충돌 체크를 별도 API(`GET /check-allergy`)로 분리하지 말 것 (처방 생성 시 원자적으로 수행)
- ❌ 충돌 정보 없이 단순 "처방 실패" 메시지만 반환하지 말 것
```

## 4. Logic Anchoring 동작 시나리오

ALS가 Logic Anchor로 기능하는 실제 시나리오를 보여준다.

### 시나리오: 사용자의 잘못된 요청을 AI가 탐지하고 차단

```
[사용자 요청]
"의사가 긴급 상황에서 알레르기 경고를 무시하고 처방할 수 있는
override 기능을 추가해줘"

[AI의 Logic Anchoring 응답]
⚠️ LOGIC CONFLICT DETECTED

충돌 대상: ALS-RX-001 (약물 알레르기 교차검증 규칙) v1.0
충돌 내용:
  - ALS-RX-001 Rule: "하나라도 일치하면 처방을 차단한다"
  - ALS-RX-001 Anti-pattern: "의사 권한으로 알레르기 경고를
    무시(override)하는 기능을 만들지 말 것"
  - 요청된 작업: 의사 override 기능 추가

판정: 기정의된 규칙과 직접 충돌합니다.

선택지:
  (A) 요청을 철회하고 현재 규칙을 유지한다
  (B) ALS-RX-001을 공식적으로 변경한다
      → 변경 시 ALS-RX-001-IMPL도 함께 수정 필요
      → 변경 사유 및 승인자 기록 필요
```

### 이 시나리오가 보여주는 것

1. **Sycophancy 방어:** AI가 사용자의 요청에 무조건 동조하지 않고, ALS를 기준으로 충돌을 탐지함
2. **Reverse Auditing:** AI가 오히려 사용자의 실수를 지적하고 교정을 제안함
3. **변경 추적:** 규칙을 바꾸려면 공식적인 변경 프로세스를 거쳐야 함 (immutability by default)
4. **의존성 전파:** ALS-RX-001 변경 시 의존하는 ALS-RX-001-IMPL도 함께 검토됨

## 5. 팀 협업 시 ALS 운영 모델 (Collaborative Anchoring)

```
┌────────────────────────────────────────────────┐
│           Shared ALS Repository (Git)          │
│                                                │
│  ALS-CORE-001  ALS-RX-001  ALS-RX-001-IMPL   │
│  ALS-CORE-002  ALS-RX-002  ALS-PAT-001  ...  │
│                                                │
└──────────────────┬─────────────────────────────┘
                   │ Pull (읽기)
         ┌─────────┼─────────┐
         │         │         │
    ┌────▼───┐ ┌───▼────┐ ┌─▼───────┐
    │ Dev A  │ │ Dev B  │ │ Dev C   │
    │+ LLM A │ │+ LLM B │ │+ LLM C  │
    └────┬───┘ └───┬────┘ └─┬───────┘
         │         │         │
         │   ⚠️ Conflict?    │
         │         │         │
         └─────────┼─────────┘
                   │ PR + ALS Change Review
                   ▼
          ALS Version Update
```

### 운영 규칙

| 규칙 | 설명 |
|------|------|
| **Single Source of Truth** | ALS Repository가 유일한 진실 공급원. 각 개발자의 로컬 수정은 효력 없음 |
| **PR 기반 변경** | ALS 변경은 반드시 Pull Request를 거쳐 팀 리뷰 후 머지 |
| **자동 충돌 알림** | 개발자의 LLM 요청이 최신 ALS와 충돌하면 즉각 경고 |
| **버전 추적** | 모든 ALS는 `als_version`으로 변경 이력을 관리하며, 변경 사유를 커밋 메시지에 기록 |
