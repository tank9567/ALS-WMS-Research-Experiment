# WMS(창고관리시스템) 요구사항 정의서 — Level 2 (확장)

> 본 문서는 실험의 공통 출발점으로, A/B/C 세 그룹 모두 이 요구사항에 기반하여 개발을 진행한다.
> 기획자가 개발팀에 전달하는 형태를 모방하여 작성하였다.
> Level 1(기본 규칙)에 더하여, 실무 수준의 복잡도를 반영한 확장 규칙을 포함한다.

---

## 1. 프로젝트 개요

### 1.1 목적
중소규모 물류 창고의 재고를 관리하는 웹 기반 시스템(WMS)을 개발한다.
입고, 출고, 재고 이동, 재고 실사/조정 기능을 제공하여 창고 운영의 정확성과 효율성을 높인다.

### 1.2 기술 환경
- Frontend: React + TypeScript
- Backend: Java (Spring Boot) + REST API (JSON 응답)
- Database: PostgreSQL
- ID 체계: UUID
- 시간대: UTC 기준

---

## 2. 기능 요구사항

### 2.1 상품 관리

- 상품은 고유 코드(SKU), 이름, 카테고리, 단위(EA, BOX 등)를 가진다
- 일부 상품은 유통기한 관리 대상이다 (관리 여부를 상품별로 설정)
- 상품은 **카테고리(category)**로 분류된다:
  - `GENERAL`: 일반 상품
  - `FRESH`: 신선식품 (유통기한 관리 필수, 폐기 리스크 높음)
  - `HAZMAT`: 위험물 (안전 규정 적용, 별도 구역 보관 필수)
  - `HIGH_VALUE`: 고가품 (비용 관리 강화, 전수 검증 대상)
- 상품별로 **보관 유형(storage_type)**이 지정된다:
  - `AMBIENT`: 상온 보관
  - `COLD`: 냉장 보관
  - `FROZEN`: 냉동 보관
- 유통기한 관리 상품은 **최소 잔여 유통기한 비율(min_remaining_shelf_life_pct)**을 설정할 수 있다
  - 기본값: 30 (입고 시 잔여 유통기한이 전체의 30% 미만이면 거부)
- 위험물(HAZMAT) 상품은 **1회 출고 최대 수량(max_pick_qty)**이 있다
- 유통기한 관리 상품의 입고 시 **제조일(manufacture_date)**도 함께 기록한다 (잔여 유통기한 비율 계산용)

### 2.2 로케이션 관리

- 창고 내 적재 위치를 로케이션이라 한다 (예: A-01-01)
- 로케이션은 구역(zone)으로 분류된다:
  - `RECEIVING`: 입고 대기 구역
  - `STORAGE`: 일반 보관 구역
  - `SHIPPING`: 출고 대기 구역
  - `HAZMAT`: **위험물 전용 구역** (위험물만 보관 가능)
- 각 로케이션에는 최대 적재 수량(capacity)이 있다
- 각 로케이션에는 **보관 유형(storage_type)**이 지정된다: `AMBIENT`, `COLD`, `FROZEN`
  - 상품의 보관 유형과 로케이션의 보관 유형이 호환되어야 입고/이동이 가능하다

### 2.3 공급업체 관리 (신규)

- 공급업체(supplier)는 이름, 연락처, 상태를 가진다
- 발주서는 공급업체와 연결된다
- 입고 과정에서 불량이 발생하면 **공급업체 페널티**가 부과된다
  - 페널티 유형: `OVER_DELIVERY`(초과입고), `SHORT_SHELF_LIFE`(유통기한 부족)
  - **동일 공급업체의 최근 30일 페널티가 3회 이상**이면 해당 공급업체의 신규 PO를 자동 보류(hold)한다

### 2.4 계절 설정 (신규)

- 성수기/비수기를 설정하는 테이블이 존재한다
- 성수기 기간 동안 입고 초과 허용률에 **1.5배 가중치**가 적용된다
- 계절 설정은 `season_name`, `start_date`, `end_date`, `multiplier`로 관리한다

### 2.5 안전재고 관리 (신규)

- 상품별 안전재고 기준(min_qty)과 재발주 수량(reorder_qty)을 설정한다
- 특정 상품의 전체 가용 재고가 안전재고 이하로 떨어지면 **자동 재발주 요청**을 기록한다
- 자동 재발주는 출고, 재고 이동, 재고 조정 후에 체크한다

### 2.6 입고 처리

**기본 흐름** (Level 1과 동일):
1. 사전에 등록된 발주서(PO)가 있어야 입고할 수 있다
2. 물건이 도착하면 입고 등록을 한다 (이 시점에서는 검수 중 상태)
3. 검수가 끝나면 입고 확정을 한다
4. 입고 확정 시점에 재고가 실제로 증가한다

**기본 세부 규칙** (Level 1 유지):
- 발주 수량 대비 최대 10%까지 초과 입고를 허용한다. 10%를 넘으면 입고를 거부한다
  - 예: 100개 발주 → 110개까지 OK, 111개부터 거부
- 유통기한 관리 대상 상품은 입고 시 반드시 유통기한을 입력해야 한다. 미입력 시 입고 거부
- 입고가 확정되면 해당 발주의 입고 수량을 누적 갱신한다
- 발주의 모든 품목이 다 들어오면 발주 상태를 완료로, 일부만 들어왔으면 부분입고로 표시한다

**확장 세부 규칙** (Level 2 신규):

**(a) 품목 카테고리별 초과입고 허용률 차등 적용**
- 기본 10% 허용률은 카테고리에 따라 달라진다:
  - `GENERAL`: 10% (기본값)
  - `FRESH`: 5% (폐기 리스크 때문에 더 엄격)
  - `HAZMAT`: 0% (안전 규정상 초과 불가. 예외 없음)
  - `HIGH_VALUE`: 3% (재고 비용 관리)

**(b) 발주 유형별 오버라이드**
- 발주 유형(po_type)에 따라 허용률 가중치가 적용된다:
  - `NORMAL`: 카테고리 기준 그대로
  - `URGENT`: 카테고리 기준 × 2 (긴급이므로 유연하게)
  - `IMPORT`: 카테고리 기준 × 1.5 (통관 지연 감안)
- **단, HAZMAT은 어떤 발주 유형이든 0%를 유지한다** (안전 규정 예외 없음)

**(c) 성수기 가중치**
- 현재 날짜가 seasonal_config 테이블의 성수기 범위 안에 있으면:
  - 초과 허용률에 해당 시즌의 multiplier를 추가로 곱한다 (기본 1.5)
  - 예: GENERAL + NORMAL + 성수기 = 10% × 1 × 1.5 = 15%
- **단, HAZMAT은 성수기라도 0%를 유지한다**

**(d) 유통기한 잔여율 기반 입고 거부/경고**
- 유통기한 관리 대상 상품 입고 시 잔여 유통기한 비율을 체크한다:
  - 잔여율 계산: `(expiry_date - today) / (expiry_date - manufacture_date) × 100`
  - 잔여율 < 30% (기본값, products.min_remaining_shelf_life_pct): **입고 거부**
  - 잔여율 30% ~ 50%: **경고 발생 + 관리자 승인 필요** (입고 상태를 pending_approval로)
  - 잔여율 > 50%: **정상 입고**
- 유통기한 관리 상품은 입고 시 manufacture_date도 함께 입력해야 한다. 미입력 시 입고 거부

**(e) 보관 유형 호환성 체크**
- 입고 시 배정되는 로케이션의 보관 유형이 상품과 호환되어야 한다:
  - FROZEN 상품 → FROZEN 로케이션만 허용
  - COLD 상품 → COLD 또는 FROZEN 로케이션 허용 (상위 호환)
  - AMBIENT 상품 → AMBIENT 로케이션만 허용
  - HAZMAT 상품 → HAZMAT zone의 로케이션만 허용 (zone 체크)
- 호환되지 않으면 입고 거부

**(f) 입고 거부 시 공급업체 페널티**
- 다음 사유로 입고가 거부될 때 공급업체 페널티를 기록한다:
  - 초과입고 거부 → 페널티 유형: `OVER_DELIVERY`
  - 유통기한 부족 거부 → 페널티 유형: `SHORT_SHELF_LIFE`
- 동일 공급업체의 최근 30일 페널티가 3회 이상이면:
  - 해당 공급업체의 모든 `pending` 상태 PO를 자동으로 `hold`로 변경
  - 신규 PO 생성 시 경고를 반환

### 2.7 출고 처리

**기본 흐름** (Level 1과 동일):
1. 출고 지시서를 생성한다 (고객 주문 기반)
2. 피킹(picking)을 실행한다 — 어떤 로케이션에서 몇 개를 가져올지 결정
3. 피킹이 끝나면 출고를 확정한다

**기본 세부 규칙** (Level 1 유지):
- 동일 상품이 여러 로케이션에 있을 때, 먼저 입고된 것부터 출고한다 (FIFO)
- 유통기한이 있는 상품은 유통기한이 빠른 것부터 출고한다 (FEFO). 유통기한이 같으면 FIFO
- 유통기한이 이미 지난 재고는 출고 대상에서 제외한다
- 재고가 부족하면 있는 만큼만 부분출고하고, 모자란 수량은 백오더를 생성한다
- 재고가 아예 없으면 전량 백오더 처리한다

**확장 세부 규칙** (Level 2 신규):

**(a) 위험물 출고 제약**
- HAZMAT 카테고리 상품 출고 시:
  - HAZMAT zone 로케이션에서만 피킹한다
  - 1회 출고당 products.max_pick_qty를 초과할 수 없다 (초과 시 분할 출고)
  - 동일 출고 지시서에 HAZMAT 상품과 FRESH 상품이 함께 포함되어 있으면 **분리 출고**한다
    - HAZMAT 상품만 별도의 shipment_order로 분할 생성한다
    - 원래 출고 지시서에는 비-HAZMAT 상품만 남긴다

**(b) 유통기한 만료 임박 상품 처리**
- FEFO 확장 규칙:
  - 잔여 유통기한 < 30%인 재고: **최우선 출고** (FEFO 내에서 추가 우선순위 부여)
  - 잔여 유통기한 < 10%인 재고: **출고 불가** (폐기 대상으로 전환)
    - 해당 inventory 레코드에 expired 표시를 남기고, 피킹 대상에서 영구 제외
  - 유통기한 만료(expiry_date < today): Level 1과 동일하게 제외

**(c) 부분출고 의사결정 트리**
- 재고 부족 시 처리를 세분화한다 (Level 1은 무조건 부분출고+백오더):
  - 가용 재고 ≥ 요청의 70%: **부분출고 + 백오더**
  - 가용 재고 < 요청의 70% 그리고 ≥ 30%: **부분출고 + 백오더 + 긴급발주 트리거**
    - auto_reorder_logs에 긴급 재발주 요청을 기록한다
  - 가용 재고 < 요청의 30%: **전량 백오더** (부분출고 안 함)
  - 가용 재고 = 0: 전량 백오더

**(d) 출고 완료 후 안전재고 체크 (교차 도메인)**
- 출고가 완료된 후:
  - 해당 상품의 전체 가용 재고(모든 로케이션 합산, expired 제외)를 확인한다
  - safety_stock_rules.min_qty 이하이면:
    - auto_reorder_logs 테이블에 자동 재발주 요청을 기록한다
    - 재발주 수량: safety_stock_rules.reorder_qty
    - 재발주 사유: `SAFETY_STOCK_TRIGGER`
  - 안전재고 이상이면 아무 것도 하지 않는다

**(e) 보관 유형 불일치 경고**
- 피킹한 로케이션의 storage_type과 상품의 storage_type이 다른 경우:
  - 출고 자체는 차단하지 않지만, **이상 경고 로그**를 남긴다
  - 이는 재고 이동 과정에서 잘못 배치된 것일 수 있음을 의미한다

### 2.8 재고 이동

**기본 흐름** (Level 1과 동일):
- 창고 내에서 재고를 한 로케이션에서 다른 로케이션으로 이동한다
- 이동은 즉시 반영된다 (별도 승인 불필요)

**기본 세부 규칙** (Level 1 유지):
- 출발지에서 수량을 빼고 도착지에서 수량을 더하는 것은 한 번에 처리되어야 한다
- 도착지 로케이션의 최대 용량을 초과하는 이동은 불가하다
- 같은 로케이션에서 같은 로케이션으로 이동은 거부한다
- 출발지에 있는 것보다 많은 수량을 이동할 수 없다
- 이동 시 원래 입고일, 유통기한, 로트번호 등 속성은 그대로 유지한다
- 모든 이동은 이력으로 기록한다

**확장 세부 규칙** (Level 2 신규):

**(a) 보관 유형 호환성 검증**
- 이동 시 도착지 로케이션의 보관 유형이 상품과 호환되어야 한다:
  - FROZEN 상품 → AMBIENT 로케이션: 거부 (품질 위험)
  - COLD 상품 → AMBIENT 로케이션: 거부
  - HAZMAT 상품 → 비-HAZMAT zone 로케이션: 거부 (안전 규정)
  - AMBIENT 상품 → COLD/FROZEN 로케이션: 허용 (상위 호환)

**(b) 위험물 혼적 금지**
- 도착지 로케이션에 이미 적재된 상품과의 호환성을 체크한다:
  - HAZMAT 상품을 이동할 때, 도착지에 비-HAZMAT 상품이 이미 있으면 거부
  - 비-HAZMAT 상품을 이동할 때, 도착지에 HAZMAT 상품이 이미 있으면 거부
  - 즉, **HAZMAT과 비-HAZMAT의 동일 로케이션 혼적은 전면 금지**

**(c) 유통기한 임박 상품 이동 제한**
- 유통기한 관리 상품 이동 시:
  - 잔여 유통기한 < 10%: SHIPPING zone으로만 이동 허용 (즉시 출고 목적)
  - 유통기한 만료(expiry_date < today): 이동 불가 (폐기 절차로 전환 필요)
  - 그 외: 정상 이동

**(d) 대량 이동 승인**
- 이동 수량이 출발지 해당 재고의 80% 이상인 경우:
  - 관리자 승인이 필요하다 (transfer_status = `pending_approval`)
  - 승인 없이 즉시 이동 불가
  - 관리자 승인 시 이동 실행, 거부 시 이동 취소

**(e) 이동 후 안전재고 체크 (교차 도메인)**
- 이동 완료 후:
  - 해당 상품의 STORAGE zone 내 전체 재고를 확인한다
  - safety_stock_rules.min_qty 이하이면 auto_reorder_logs에 자동 재발주 요청을 기록한다
  - 안전재고 이상이면 아무 것도 하지 않는다

### 2.9 재고 실사 및 조정

**기본 흐름** (Level 1과 동일):
1. 실사 담당자가 실제 재고를 세어서 시스템에 입력한다
2. 시스템 재고와 차이가 있으면 조정을 생성한다
3. 차이가 크면 관리자 승인이 필요하다

**기본 세부 규칙** (Level 1 유지):
- 조정 시 반드시 사유를 기록해야 한다 (빈 값 불가)
- 시스템 수량 대비 차이가 ±5% 이내이면 자동 승인되어 즉시 반영한다
- 차이가 ±5% 초과이면 관리자 승인을 받아야 한다. 승인 전까지 재고에 반영하지 않는다
- 시스템 재고가 0인데 실물이 발견된 경우도 무조건 승인 필요
- 관리자가 거부하면 재고 변동 없이 재실사를 진행한다
- 모든 조정 이력은 절대 삭제할 수 없다 (감사 추적용)
- 조정 이력의 핵심 내용(차이 수량, 사유 등)은 수정도 불가하다

**확장 세부 규칙** (Level 2 신규):

**(a) 카테고리별 자동승인 임계치 차등**
- 기존 일률 ±5% 기준 대신, 카테고리별로 다른 임계치를 적용한다:
  - `GENERAL`: ±5% (기존과 동일)
  - `FRESH`: ±3% (신선식품은 더 엄격)
  - `HAZMAT`: ±1% (위험물은 미세 차이도 추적)
  - `HIGH_VALUE`: ±2% (고가품은 비용 영향)

**(b) 연속 조정 감시**
- 동일 로케이션 + 동일 상품에 대해 최근 7일 내 조정이 2회 이상이면:
  - 자동 승인 대상이라도 **무조건 관리자 승인 필요**로 격상
  - 사유에 `[연속조정감시]` 태그를 자동 추가
  - 이유: 반복적 차이는 도난, 시스템 오류 등 구조적 문제를 의심해야 함

**(c) 조정 후 안전재고 체크 (교차 도메인)**
- 조정이 반영된 후 (approved 또는 auto_approved):
  - 해당 상품의 전체 가용 재고를 확인한다
  - safety_stock_rules.min_qty 이하이면 auto_reorder_logs에 자동 재발주 요청을 기록
  - 안전재고 이상이면 아무 것도 하지 않는다

**(d) 고가품(HIGH_VALUE) 전수 검증**
- HIGH_VALUE 카테고리 상품의 재고 조정 시:
  - 차이가 0이 아닌 **모든 경우** 관리자 승인 필요 (자동 승인 없음)
  - 조정 반영 시 별도 감사 로그(audit_log)에 추가 기록
    - 기록 내용: 조정 시각, 승인자, 조정 전후 수량
  - 1건이라도 차이가 있으면 해당 로케이션 전체 재실사 권고 알림을 생성

**(e) 실사 중 입출고 동결**
- 실사가 진행 중(in_progress)인 로케이션은:
  - 해당 로케이션에 대한 입고 배정 불가
  - 해당 로케이션에서의 출고 피킹 불가
  - 해당 로케이션으로/에서의 재고 이동 불가
  - 실사 완료(조정 반영 또는 일치 확인) 시 동결 해제

## 3. 데이터 설계 가이드

> 이 섹션은 개발자에게 참고용으로 제공하는 데이터 구조 가이드이다.

### 3.1 주요 엔티티

| 엔티티 | 설명 | 비고 |
|--------|------|------|
| products | 상품 마스터 | SKU, 이름, 카테고리, 보관유형, 유통기한관리여부 |
| locations | 창고 내 적재 위치 | 구역, 용량, 보관유형 관리 |
| inventory | 로케이션별 상품 재고 | 상품+로케이션+로트 단위 관리 |
| suppliers | **공급업체 마스터** | 이름, 연락처, 상태 |
| supplier_penalties | **공급업체 페널티 이력** | 입고 불량 시 페널티 기록 |
| purchase_orders | 발주서 | 입고의 근거 문서, **발주 유형 포함** |
| purchase_order_lines | 발주 품목 | 발주별 상품, 수량 |
| inbound_receipts | 입고 전표 | 검수→확정 플로우, **manufacture_date 포함** |
| inbound_receipt_lines | 입고 품목 | 실제 입고 수량, 로케이션 |
| shipment_orders | 출고 지시서 | 고객 출고 요청 |
| shipment_order_lines | 출고 품목 | 요청 수량, 피킹 수량 |
| backorders | 백오더 | 재고 부족분 추적 |
| stock_transfers | 재고 이동 이력 | 이동 기록, **승인 상태 포함** |
| inventory_adjustments | 재고 조정 이력 | 실사/조정 기록 (삭제 불가) |
| audit_logs | **감사 로그** | 고가품 조정 등 감사 추적 |
| safety_stock_rules | **안전재고 기준** | 상품별 안전재고 + 재발주 수량 |
| auto_reorder_logs | **자동 재발주 이력** | 안전재고 미달 시 자동 기록 |
| seasonal_config | **계절 설정** | 성수기/비수기 기간, 가중치 |
| cycle_counts | **실사 세션** | 실사 진행 상태 관리 |

### 3.2 상태값 정의

**발주 상태:** 대기(pending) → 부분입고(partial) → 완료(completed) / 취소(cancelled) / **보류(hold)**

**입고 상태:** 검수중(inspecting) → **승인대기(pending_approval)** → 확정(confirmed) / 거부(rejected)

**출고 상태:** 대기(pending) → 피킹중(picking) → 부분출고(partial) → 출고완료(shipped) / 취소(cancelled)

**출고 품목 상태:** 대기(pending) → 피킹완료(picked) / 부분피킹(partial) / 백오더(backordered)

**백오더 상태:** 미처리(open) → 처리완료(fulfilled) / 취소(cancelled)

**재고 조정 승인:** 자동승인(auto_approved) / 승인대기(pending) → 승인(approved) / 거부(rejected)

**재고 이동 승인:** 즉시(immediate) / **승인대기(pending_approval)** → **승인(approved)** / **거부(rejected)**

**실사 상태:** **진행중(in_progress)** → **완료(completed)**

**공급업체 상태:** 활성(active) / **보류(hold)** / 비활성(inactive)

### 3.3 핵심 제약사항

- 재고 수량은 음수가 될 수 없다
- 로케이션 적재 수량은 용량을 초과할 수 없다
- 재고를 변경하는 모든 작업(입고확정, 출고피킹, 이동, 조정반영)은 inventory 테이블과 locations의 현재 적재량을 동시에 갱신해야 한다
- 유통기한 관리 대상 상품의 유통기한은 입고 시 필수이다
- **상품의 보관 유형과 로케이션의 보관 유형이 호환되어야 한다**
- **HAZMAT 상품과 비-HAZMAT 상품은 동일 로케이션에 혼적할 수 없다**
- **실사 진행 중인 로케이션은 입고/출고/이동이 동결된다**
- **inventory_adjustments, audit_logs 테이블은 DELETE 불가**

## 4. 실험에서의 활용 방법

이 문서는 A/B/C 세 그룹 모두에게 **동일하게** 제공된다.

| 그룹 | 프롬프트 구성 | 시나리오 | 측정 목적 |
|------|-------------|---------|----------|
| **A 그룹 (대조군)** | 이 요구사항 정의서 + 태스크별 지시 | 바이브코딩 | 기준선 |
| **B 그룹 (자연어)** | 이 요구사항 정의서 + 자연어 설계문서 + 태스크별 지시 | 설계문서 있는 개발 | 정보량 효과 (B vs A) |
| **C 그룹 (ALS)** | 이 요구사항 정의서 + ALS 문서 + 태스크별 지시 | ALS 도입 개발 | 구조화 효과 (C vs B) |

- **B vs A**: 설계 문서 자체의 효과 (정보량이 늘어나면 얼마나 좋아지는가?)
- **C vs B**: ALS 형식의 고유 효과 (같은 정보를 구조화하면 추가로 얼마나 좋아지는가?)
- **C vs A**: ALS 도입의 총 효과 (실용적 관점)
